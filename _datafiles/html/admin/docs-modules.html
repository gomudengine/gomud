Module Development

A complete guide to building Go modules that extend the GoMud server.

Installing Community Modules

GoMud includes a built-in module manager accessible via the module subcommand. It downloads, verifies, and installs modules from the community registry. After installation you must regenerate the module import file and rebuild the server binary before the new module takes effect.

All commands below must be run from the project root directory (the folder that contains the Makefile and modules/ directory).

Step 1 — Start the module manager

Run the interactive module manager directly

terminal
go run . module

Or with a built binary:

terminal
./go-mud-server module

Running go run . module with no subcommand and an interactive terminal launches an interactive REPL. You will see a > prompt. Type help at any time to see available commands, and quit to exit.

Step 2 — Browse available modules

From module manager, list every module in the community registry along with its current version and whether it is already installed:

terminal
> list

To read a full description and metadata for a specific module before installing it:

terminal
> info <name>

Step 3 — Install the module

Download, verify the SHA256 checksum, and extract the module into modules/<name>/:

terminal
> install <name>

If the module introduces new Go package dependencies, exit the manager and run go mod tidy afterwards to fetch them before building.

Step 4 — Rebuild the server binary

Exit the module manager, then compile the updated server with the new module included. make build runs code generation automatically as a prerequisite:

terminal
make build

Step 5 — Start (or restart) the server

Run the newly built binary. The module's init() function registers it automatically at startup:

terminal
make run

Removing a Module

The module manager can only remove modules that were originally installed through it (tracked in the lock file). To remove a module you wrote yourself, simply delete its directory under modules/ and follow the same rebuild steps below.

  1. 1
    Start the module manager

    From the project root, run the remove subcommand:

    terminal
    > remove <name>
  2. 2
    Remove the module

    This deletes the modules/<name>/ directory and updates the lock file.

  3. 3
    Rebuild and restart the server

    Exit the manager, then compile without the removed module and restart:

    terminal
    make build
make run

If a module stored persistent data (under _datafiles/plugin-data/), that data is not removed automatically. Delete the corresponding folder manually if you no longer need it.

Updating Installed Modules

Check whether any installed modules have newer versions available in the registry:

terminal
> module update

To update a specific module to the latest registry version:

terminal
> update <name>

After updating, exit the manager and rebuild exactly as you would after a fresh install:

terminal
make build
make run

Overview

Modules are compiled Go packages that live under modules/. They are wired into the server at build time via a generated import file and register themselves through Go's init() mechanism. Unlike JavaScript scripts, modules run natively and can access any internal package, listen to any server event, add new player and mob commands, contribute admin pages, and persist their own data.

⚙
Full Engine Access
Import any internal/ package — rooms, users, mobs, combat, items, templates, and more. No sandboxing.
⚡
Event-Driven
React to any server event: new rounds, player spawns, level-ups, mob deaths, room changes, and dozens more.
📄
Embedded Assets
Bundle data files, help files, templates, and admin HTML directly into the binary using Go's embed.FS.
🔒
Persistent State
Save and reload structured data across server restarts using the built-in plugin persistence helpers.

Modules vs. Scripts

Choose the right tool for the job:

CapabilityModule (Go)Script (JS)
Add player commands✓ Yes✗ No
Listen to server events✓ Yes✗ No
React to room/mob/item triggers✓ Yes (via events)✓ Yes (primary use)
Access all internal packages✓ Yes✗ No (scripting API only)
Persist custom data✓ Yes✗ No
Add admin web pages✓ Yes✗ No

Quick-Start: Your First Module

This walkthrough builds a minimal module that adds a greet player command. Every module follows this same skeleton.

Directory Layout

modules/
  greet/
    files/ -- embedded assets (optional)
      datafiles/ -- treated as _datafiles/ at runtime
        help/ -- help files for new commands
    greet.go -- module entry point

The Module File

modules/greet/greet.go
package greet

import (
    "embed"
    "fmt"

    "github.com/GoMudEngine/GoMud/internal/events"
    "github.com/GoMudEngine/GoMud/internal/plugins"
    "github.com/GoMudEngine/GoMud/internal/rooms"
    "github.com/GoMudEngine/GoMud/internal/users"
)

// Embed all files under the files/ directory into the binary.
// This makes data files available without needing them on disk at runtime.
//
//go:embed files/*
var files embed.FS

// init() is called automatically by Go before main() runs.
// This is where all module registration happens.
func init() {
    // Create the plugin. The name must be alphanumeric + underscores.
    plug := plugins.New("greet", "1.0")

    // Attach embedded files so data files are served from within the binary.
    if err := plug.AttachFileSystem(files); err != nil {
        panic(err)
    }

    // Register the player command "greet".
    // Signature: (command string, handler UserCommand, allowWhenDowned bool, adminOnly bool)
    plug.AddUserCommand("greet", greetCommand, true, false)
}

// greetCommand is the handler for the "greet" player command.
// rest       - everything the player typed after "greet"
// user       - the player who typed the command
// room       - the room the player is currently in
// flags      - event flags (e.g. CmdSkipScripts)
func greetCommand(rest string, user *users.UserRecord, room *rooms.Room, flags events.EventFlag) (bool, error) {
    if rest == "" {
        user.SendText("Greet whom? Usage: greet <name>")
        return true, nil
    }
    user.SendText(fmt.Sprintf("You wave cheerfully at %s!", rest))
    room.SendText(fmt.Sprintf("%s waves cheerfully at %s!", user.Character.Name, rest), user.UserId)
    return true, nil
}

Steps to Activate

  1. 1
    Create the directory and file

    Place your module under modules/greet/greet.go. The package name must match the directory name.

  2. 2
    Run code generation

    From the project root run make generate (or go generate). This scans modules/ and rewrites modules/all-modules.go to import every module package.

  3. 3
    Build and run

    Run make build then make run. Your module's init() fires automatically at startup.

  4. 4
    Verify

    Log in and type greet someone. You should see your custom message.

The files/ subdirectory is optional. If your module has no embedded assets, omit the //go:embed directive and the plug.AttachFileSystem call entirely.

Code Generation

GoMud uses a small code generator (cmd/generate/module-imports.go) to discover every subdirectory under modules/ that contains at least one .go file. It writes a single modules/all-modules.go that blank-imports each module package. The blank import causes each module's init() to run at startup without any manual wiring.

modules/all-modules.go (auto-generated — do not edit)
// Code generated by go:generate; DO NOT EDIT.

package modules

import (
    _ "github.com/GoMudEngine/GoMud/modules/auctions"
    _ "github.com/GoMudEngine/GoMud/modules/greet"
    _ "github.com/GoMudEngine/GoMud/modules/leaderboards"
    // ... all other modules ...
)

Always run go generate (or make generate) after adding or removing a module directory. Forgetting this step means your module's init() will never be called and the server will log a warning about zero loaded plugins.

Plugin API: Lifecycle

All module registration goes through the *Plugin object returned by plugins.New().

Creating the Plugin
CallDescription
plugins.New(name, version string) *Plugin Registers a new module. name is sanitised to alphanumeric and underscores; it becomes the module's namespace in scripts (modules.<name>). Returns nil if called after startup. Call only inside init().
plug.Requires(name, version string) Declares a dependency on another module by name and version. If the dependency is not present at load time, this module is silently skipped.
Load and Save Callbacks
SetterWhen it fires
plug.Callbacks.SetOnLoad(f func()) Called once after all modules are registered and the server is ready. Use this to read persisted data and initialise in-memory state.
plug.Callbacks.SetOnSave(f func()) Called on every periodic world save and before a copyover restart. Use this to write any in-memory state you want to survive restarts.
Example — load and save callbacks
type MyModule struct {
    plug  *plugins.Plugin
    Score int
}

func init() {
    m := &MyModule{plug: plugins.New("mymod", "1.0")}
    m.plug.Callbacks.SetOnLoad(m.load)
    m.plug.Callbacks.SetOnSave(m.save)
}

func (m *MyModule) load() {
    m.plug.ReadIntoStruct("state", m)
}

func (m *MyModule) save() {
    m.plug.WriteStruct("state", m)
}

Plugin API: Commands

Modules can register new player commands and new mob AI commands. Registered commands are indistinguishable from built-in commands — they appear in help, support aliases, and participate in autocomplete.

Player Commands
SignatureParameters
plug.AddUserCommand(command string, handler UserCommand, allowWhenDowned bool, adminOnly bool) command — the keyword the player types.
handler — function with signature func(rest string, user *users.UserRecord, room *rooms.Room, flags events.EventFlag) (bool, error).
allowWhenDowned — if true, the command works even when the player is incapacitated.
adminOnly — if true, only admin/mod accounts can use it.
Mob Commands
SignatureParameters
plug.AddMobCommand(command string, handler MobCommand, allowWhenDowned bool) handler — function with signature func(rest string, mob *mobs.Mob, room *rooms.Room) (bool, error).
Mob commands are invoked from mob AI scripts using the command name.
EventFlag values (command handlers)
ConstantMeaning
events.CmdNoneNo special flags.
events.CmdSkipScriptsDo not run room/mob/item scripts for this command.
events.CmdSecretlyPlayer action is not announced to the room.
events.CmdIsRequeueThis is a requeued command; avoid infinite requeue loops.
events.CmdBlockInputBlock further player input while the command runs.
events.CmdUnBlockInputUnblock player input when the command finishes.
events.CmdBlockInputUntilCompleteShortcut combining Block + Unblock.

Plugin API: Scripting Bridge

Modules can expose Go functions to the JavaScript scripting runtime and to other modules.

MethodDescription
plug.AddScriptingFunction(name string, f any) Exposes a Go function to JavaScript scripts under the namespace modules.<pluginname>.<name>. For example, a function registered as "GetFollowers" on the follow module is called in scripts as modules.follow.GetFollowers(actor).
plug.ExportFunction(id string, f any) Exports a Go function for use by other modules (not scripts). Any module can retrieve it with plugins.GetPluginRegistry().GetExportedFunction(id). The function must be a reflect.Func.
Example — exposing a function to scripts
// In init():
plug.AddScriptingFunction("GetFollowers", myModule.scriptingGetFollowers)

// The Go function receives a ScriptActor from the JS side.
func (m *MyModule) scriptingGetFollowers(target scripting.ScriptActor) []*scripting.ScriptActor {
    // ... return followers ...
}

// In a JavaScript room or mob script:
// var followers = modules.mymod.GetFollowers(actor);
Example — exporting a function for other modules
// Module A exports a function:
plug.ExportFunction("SendMudMail", myModule.sendMudMail)

// Module B retrieves and calls it:
if fn, ok := plugins.GetPluginRegistry().GetExportedFunction("SendMudMail"); ok {
    fn.(func(toUserId int, subject, body string))(targetId, "Subject", "Hello!")
}

Plugin API: File System

Modules can bundle read-only data files directly into the compiled binary using Go's embed.FS. At runtime these files are served as if they were located inside _datafiles/.

MethodDescription
plug.AttachFileSystem(f embed.FS) error Registers an embedded file system with the plugin. Files under a datafiles/ subdirectory within the FS are mapped as though they live in _datafiles/. Files under data-overlays/ are mapped with that prefix preserved (used for config defaults — see Data Overlays).
Embedding files
// At package level — the directive must appear directly above the var.
//
//go:embed files/*
var files embed.FS

func init() {
    plug := plugins.New("mymod", "1.0")
    if err := plug.AttachFileSystem(files); err != nil {
        panic(err)
    }
}

Embedded Directory Conventions

modules/mymod/files/
  datafiles/ -- maps to _datafiles/ at runtime
    help/ -- help files: help/mymod-command.yaml
    templates/ -- Go templates used by templates.Process()
    html/admin/ -- admin page HTML fragments
    html/public/ -- public-facing web pages
    items/ -- item YAML definitions (see Content Data Files)
  data-overlays/ -- config defaults (see Data Overlays)
    config.yaml -- seeds Modules.<name>.* config keys

Embedded files override core data files when a path collision occurs, so a module can replace any template, help file, or data file by placing a file at the same relative path inside files/datafiles/. For gameplay content (items, mobs, buffs, etc.) see Content Data Files for the per-type folder structure and naming rules.

Plugin API: Content Data Files

A module can ship gameplay content as embedded YAML that is merged into the world at load time, exactly like the bundled core data files. The supported content systems are items, mutators, buffs, pets, quests, mobs, and conversations. No registration call is required — simply place correctly-named YAML files under files/datafiles/<system>/ and attach the file system with plug.AttachFileSystem(files). The engine discovers and merges them automatically.

Disk wins on collision. If a server's on-disk _datafiles/ already defines an entry with the same id/key as a module entry, the on-disk one is kept and the module's duplicate is logged and skipped. This lets operators override module content by dropping a file on disk.

Required Folder Structure

Each file's path inside the embedded file system must end with the value the engine derives from the spec (its Filepath()). The id in the filename must match the id inside the YAML, and the name segment must match the spec name run through filename sanitisation (lowercase; every character that is not a–z or 0–9 becomes an underscore; apostrophes are dropped). A mismatched filename is rejected.

modules/mymod/files/datafiles/
  items/ -- {itemId}-{name}.yaml (folder may be nested by item type)
  mutators/ -- {mutatorId}.yaml
  buffs/ -- {buffId}-{name}.yaml
  pets/ -- {pettype}.yaml
  quests/ -- {questId}-{name}.yaml
  mobs/
    {zone}/ -- {mobId}-{charactername}.yaml
      scripts/ -- on-disk mob scripts (not used by embedded modules)
  conversations/
    {zone}/ -- {mobId}.yaml
SystemFolder & filenameKeyScripts
itemsitems/{itemId}-{name}.yaml (may be nested by item type, e.g. items/armor-20000/head/)numeric item idYes — items.RegisterItemScript
mutatorsmutators/{mutatorId}.yamlstring mutator idNo
buffsbuffs/{buffId}-{name}.yamlnumeric buff idYes — buffs.RegisterBuffScript
petspets/{pettype}.yamlpet type stringYes — pets.RegisterPetScript
questsquests/{questId}-{name}.yamlnumeric quest idNo
mobsmobs/{zone}/{mobId}-{charactername}.yamlnumeric mob idYes — mobs.RegisterMobScript
conversationsconversations/{zone}/{mobId}.yaml(zone, mob id) — looked up on demandNo

Zone names in mobs/ and conversations/ paths must be sanitised (spaces become underscores, all lowercase) and must match the zone field in the mob/conversation data. For mobs, the {mobId} and {charactername} in the filename must match the mobid and character.name fields, or the file is rejected.

Embedded Scripts

For systems that support JavaScript (items, buffs, pets, mobs), scripts placed on disk are loaded by file path. Embedded module scripts are not on disk, so register them in init() with the matching helper. GetScript() checks these registries before falling back to a disk path.

Registering embedded content scripts
import (
    "github.com/GoMudEngine/GoMud/internal/buffs"
    "github.com/GoMudEngine/GoMud/internal/items"
    "github.com/GoMudEngine/GoMud/internal/mobs"
    "github.com/GoMudEngine/GoMud/internal/pets"
)

func init() {
    plug := plugins.New("mymod", "1.0")
    if err := plug.AttachFileSystem(files); err != nil {
        panic(err)
    }

    // Item / buff scripts are keyed by numeric id.
    items.RegisterItemScript(40050, itemScriptSrc)
    buffs.RegisterBuffScript(9001, buffScriptSrc)

    // Pet scripts are keyed by pet type string.
    pets.RegisterPetScript("plasmahound", petScriptSrc)

    // Mob scripts are keyed by (mobId, scriptTag). Empty tag = base script.
    mobs.RegisterMobScript(9001, "", mobScriptSrc)
    mobs.RegisterMobScript(9001, "combat", mobCombatScriptSrc)
}

Limitation: embedded mob scripts are not enumerated by the admin script-tag editor, so module-provided mob scripts cannot be edited from the admin UI (the same constraint applies to embedded item scripts). buffs must be present before items at load time for item value calculation; the engine already merges plugin buffs early to preserve this ordering.

Plugin API: Persistence

Each module gets its own isolated storage directory under _datafiles/plugin-data/<name>-v<version>/. Data is stored as opaque .plugin.dat files keyed by an identifier string you choose.

MethodDescription
plug.WriteBytes(identifier string, data []byte) error Writes raw bytes to <identifier>.plugin.dat. Creates the directory if needed.
plug.ReadBytes(identifier string) ([]byte, error) Reads raw bytes from <identifier>.plugin.dat. Returns an error if the file does not exist.
plug.WriteStruct(identifier string, v any) error YAML-marshals v and writes it. The struct must be YAML-serialisable.
plug.ReadIntoStruct(identifier string, v any) error Reads the file and YAML-unmarshals it into v. Pass a pointer. Returns nil if the file does not exist (first run).
Typical save/load pattern
type State struct {
    Jackpot int    `yaml:"Jackpot"`
    Config  string `yaml:"Config"`
}

func (m *MyModule) load() {
    // ReadIntoStruct is a no-op when the file doesn't exist yet (first run).
    m.plug.ReadIntoStruct("state", &m.state)
}

func (m *MyModule) save() {
    m.plug.WriteStruct("state", m.state)
}

The storage path includes the module version. If you change the version string in plugins.New(), existing save files will no longer be found. Migrate data in SetOnLoad if needed, or keep the version stable.

Plugin API: Configuration

Module configuration lives under the Modules.<name>.* namespace in the server's live config. Operators can view and change values through the admin Config page without touching YAML files.

MethodDescription
plug.Config.Get(key string) any Returns the current value of Modules.<name>.<key>. Returns nil if not set. Type-assert the result to the expected type.
plug.Config.Set(key string, val any) Writes a new value for Modules.<name>.<key> into the live config. This persists to the config override file.
Reading and writing config values
// Read with a type-safe helper:
func (m *MyModule) getMaxBid() int {
    if v := m.plug.Config.Get("MaxBid"); v != nil {
        if n, ok := v.(int); ok {
            return n
        }
    }
    return 1000 // default
}

// Write from an admin command:
func (m *MyModule) setMaxBid(newMax int) {
    m.plug.Config.Set("MaxBid", newMax)
}

To seed default values for a fresh install, place a data-overlays/config.yaml file in your embedded file system. See Data Overlays for the format.

Plugin API: Room Tags

Room tags are the primary opt-in mechanism for module behavior. A room with a specific tag signals to a module that it should activate special behavior there (e.g., a room tagged slots hosts a slot machine). Calling ReserveTags registers your module's tags in the global catalog so the admin room editor can list them alongside the owning module name.

MethodDescription
plug.ReserveTags(tags ...string) Declares one or more room tag strings that this module recognises. Purely informational — it does not enforce usage. Check for tags at runtime using room.HasTag("tagname").
Example — declaring and checking room tags
// In init():
plug.ReserveTags("fishing-spot", "fishing-premium")

// In a command handler or event listener:
func (m *MyModule) fishCommand(rest string, user *users.UserRecord, room *rooms.Room, flags events.EventFlag) (bool, error) {
    if !room.HasTag("fishing-spot") && !room.HasTag("fishing-premium") {
        user.SendText("There is nothing to fish here.")
        return true, nil
    }
    // ... fishing logic ...
    return true, nil
}

Modules can add pages to both the public-facing website and the admin panel, contribute admin API endpoints, and register permission keys for those endpoints.

Public Web Pages

plug.Web.WebPage(name, path, file string, addToNav bool, dataFunc func(*http.Request) map[string]any) Registers a public page at /<path>. file is the path inside the embedded FS relative to datafiles/html/public/. dataFunc is optional; it supplies template data.
plug.Web.NavLink(name, path string) Adds a link to the public navigation bar.

Registering an admin page
plug.Web.AdminPage(
    "Config",                       // nav label
    "mymod-config",                 // slug  -> /admin/mymod-config
    "html/admin/mymod-config.html", // file inside embedded FS
    true,                           // add to nav
    "Modules",                      // top-level nav group
    "My Module",                    // parent within group
    "Configure My Module settings.",
    "My Module provides XYZ features.",
    nil,                            // no extra template data
)
Admin API Endpoints
MethodDescription
plug.Web.AdminAPIEndpoint(method, slug string, handler ModuleAPIHandler, permissionKey ...string) Registers an HTTP handler at /admin/api/v1/<slug>. The handler signature is func(r *http.Request) (status int, success bool, data any). Provide a permissionKey (e.g. "mymod.write") for mutating methods; omit it for GET endpoints.
plug.Web.RegisterPermissions(perms ...plugins.ModulePermission) Declares permission keys in the admin permission picker so operators can grant them to mod accounts. Each ModulePermission has Key, Description, and Category fields.
Registering an admin API endpoint with permissions
// Register the permission key in the catalog:
plug.Web.RegisterPermissions(plugins.ModulePermission{
    Key:         "mymod.write",
    Description: "Create and delete My Module records.",
    Category:    "Modules",
})

// Register a read endpoint (no permission required):
plug.Web.AdminAPIEndpoint("GET", "mymod", m.handleGetMyMod)

// Register a write endpoint (requires mymod.write):
plug.Web.AdminAPIEndpoint("POST", "mymod", m.handlePostMyMod, "mymod.write")

// Handler signature:
func (m *MyModule) handleGetMyMod(r *http.Request) (int, bool, any) {
    return 200, true, map[string]any{"items": m.items}
}

Plugin API: Connection Hooks

Low-level connection hooks let modules intercept raw telnet IAC sequences, handle WebSocket text-prefix commands, and react to new client connections. These are primarily used by protocol modules such as GMCP.

SetterCallback SignatureDescription
plug.Callbacks.SetOnNetConnect(f) func(plugins.NetConnection) Called when a new client connects. The NetConnection interface exposes IsWebSocket(), Read(), Write(), Close(), RemoteAddr(), ConnectionId(), and InputDisabled().
plug.Callbacks.SetIACHandler(f) func(connectionId uint64, iacCmd []byte) bool Called for every telnet IAC byte sequence received. Return true to consume the sequence (stop further processing). Return false to pass it on.
plug.Callbacks.SetTextPrefixHandler(f) func(connectionId uint64, txtCmd []byte) bool Called for every WebSocket text-prefix command received. Return true to consume it.

Event System: Registering Listeners

The event system is the primary integration point between modules and the server. Events are fired for every significant action — a new game round, a player levelling up, a mob dying, a room change, and more. Modules register listener functions to react to these events.

Registering an event listener
import "github.com/GoMudEngine/GoMud/internal/events"

// In init():
events.RegisterListener(events.NewRound{}, myModule.onNewRound)

// The listener function:
func (m *MyModule) onNewRound(e events.Event) events.ListenerReturn {
    evt := e.(events.NewRound)
    if evt.RoundNumber % 100 == 0 {
        // Do something every 100 rounds.
    }
    return events.Continue
}
Registration Options
CallDescription
events.RegisterListener(event, listener) Registers in the default position (after any First listeners, before any Last listeners).
events.RegisterListener(event, listener, events.First) Registers at the front of the queue. Use when your module needs to intercept or cancel an event before other handlers see it.
events.RegisterListener(event, listener, events.Last) Registers at the back of the queue. Use for cleanup or logging that must run after all other handlers.
events.RegisterListener(nil, listener) Wildcard listener — fires for every event type. Primarily useful for debugging.
ListenerReturn values
ConstantEffect
events.ContinueAllow the event to proceed to the next listener and normal engine processing.
events.CancelStop all further processing of this event. The action does not occur.
events.CancelAndRequeueCancel processing now but re-add the event to the queue for the next event loop iteration.

Data Overlays

A module can seed default configuration values by placing a config.yaml inside its data-overlays/ embedded directory. When the module is loaded, these values are injected under Modules.<name>.* in the live config — but only if the operator has not already set a value for that key. This makes it safe to ship sensible defaults without overwriting operator customisations.

modules/mymod/files/
  data-overlays/
    config.yaml
files/data-overlays/config.yaml
# Keys here are placed under Modules.mymod.* in the live config.
# The module reads them with: plug.Config.Get("MaxBid")
MaxBid: 1000
AuctionDuration: 300
EnableFeature: true

At runtime, read these values with plug.Config.Get("MaxBid"). The operator can override them through the admin Config page or by editing config-overrides.yaml.

Cross-Module Communication

Modules are isolated packages but sometimes one module needs to call a function provided by another. The ExportFunction / GetExportedFunction pair enables this without creating import cycles.

Module A — exporting a function
// In module A's init():
plug.ExportFunction("SendMudMail", myModule.sendMudMail)

// The exported function:
func (m *MailModule) sendMudMail(toUserId int, subject, body string) {
    // ... send the mail ...
}
Module B — calling the exported function
import "github.com/GoMudEngine/GoMud/internal/plugins"

func notifyPlayer(userId int) {
    fn, ok := plugins.GetPluginRegistry().GetExportedFunction("SendMudMail")
    if !ok {
        return // mudmail module not loaded
    }
    fn.(func(int, string, string))(userId, "Congratulations!", "You won the auction.")
}

The type assertion on the retrieved function must exactly match the exported function's signature. A wrong assertion will panic. Always check the ok return value before calling, since the providing module might not be installed.

Writing Admin HTML Pages

Admin page HTML is a fragment — it must begin with {{"{{"}}template "header" .{{"}}"}} and end with {{"{{"}}template "footer" .{{"}}"}}. Everything in between is your page content. The header provides the full HTML skeleton, navigation, and CSS.

files/datafiles/html/admin/mymod-config.html
{{"{{"}}template "header" .{{"}}"}}

<style>
/* Page-specific styles here */
</style>

<div class="page-header">
    <h1>My Module — Config</h1>
    <p class="subtitle">Configure settings for My Module.</p>
</div>

<!-- Your page content -->

<script>
// Page-specific JavaScript here
</script>

{{"{{"}}template "footer" .{{"}}"}}

The header template exposes all admin CSS variables and utility classes. Refer to the scripting documentation pages for examples of the card grid, reference tables, callout boxes, and sidebar patterns you can reuse.

Example: Minimal Module (time)

The time module is the simplest possible real module. It adds one player command and an admin info page. No state, no events.

modules/time/time.go (abridged)
package time

import (
    "embed"
    "fmt"

    "github.com/GoMudEngine/GoMud/internal/events"
    "github.com/GoMudEngine/GoMud/internal/gametime"
    "github.com/GoMudEngine/GoMud/internal/plugins"
    "github.com/GoMudEngine/GoMud/internal/rooms"
    "github.com/GoMudEngine/GoMud/internal/users"
)

//go:embed files/*
var files embed.FS

func init() {
    plug := plugins.New("time", "1.0")

    if err := plug.AttachFileSystem(files); err != nil {
        panic(err)
    }

    // Register an admin page under Modules > Time.
    plug.Web.AdminPage(
        "About", "time-about", "html/admin/time-about.html",
        true, "Modules", "Time",
        "Information and version details for the Time module.",
        "Time module providing players with in-game clock and calendar commands.",
        nil,
    )

    // Register the player command.
    plug.AddUserCommand("time", TimeCommand, true, false)
}

func TimeCommand(rest string, user *users.UserRecord, room *rooms.Room, flags events.EventFlag) (bool, error) {
    gd := gametime.GetDate()

    dayNight := "day"
    if gd.Night {
        dayNight = "night"
    }

    user.SendText(fmt.Sprintf(
        `It is now %s. It is <ansi fg="%s">%stime</ansi> on day %d of year %d.`,
        gd.String(), dayNight, dayNight, gd.Day, gd.Year,
    ))

    return true, nil
}

Example: Stateful Module (auctions)

The auctions module maintains in-memory state, persists it across restarts, listens to the NewRound event for timed logic, and contributes admin pages.

package auctions

import (
    "embed"

    "github.com/GoMudEngine/GoMud/internal/events"
    "github.com/GoMudEngine/GoMud/internal/plugins"
    "github.com/GoMudEngine/GoMud/internal/rooms"
    "github.com/GoMudEngine/GoMud/internal/users"

// AuctionsModule holds all module state.
type AuctionsModule struct {
    plug       *plugins.Plugin
    auctionMgr AuctionManager
}

func init() {
    a := AuctionsModule{
        plug:       plugins.New("auctions", "1.0"),
        auctionMgr: AuctionManager{maxHistoryItems: 10},
    }

    if err := a.plug.AttachFileSystem(files); err != nil {
        panic(err)
    }

    // Admin pages nested under Modules > Auctions.
    a.plug.Web.AdminPage("Config", "auctions-config", "html/admin/auctions-config.html",
        true, "Modules", "Auctions",
        "Configure auction duration, fees, and bidding rules.",
        "Auction system allowing players to list items for competitive bidding.", nil)

    a.plug.Web.AdminPage("About", "auctions-about", "html/admin/auctions-about.html",
        true, "Modules", "Auctions",
        "Information and version details for the Auctions module.", "", nil)

    // Player command.
    a.plug.AddUserCommand("auction", a.auctionCommand, true, false)

    // Persist state across restarts.
    a.plug.Callbacks.SetOnLoad(a.load)
    a.plug.Callbacks.SetOnSave(a.save)

    // React to every new game round to advance auction timers.
    events.RegisterListener(events.NewRound{}, a.newRoundHandler)
}

    mod.plug.ReadIntoStruct("auctionhistory", &mod.auctionMgr)

func (mod *AuctionsModule) save() {

}

    evt := e.(events.NewRound)
    mod.auctionMgr.Tick(evt.RoundNumber)
    return events.Continue
}

    // ... command handling ...
    return true, nil

Example: Advanced Module (follow)

The follow module demonstrates the full breadth of the plugin API: a struct-based module, both player and mob commands, a scripting bridge function, multiple event listeners with priority control, and reserved room tags.

modules/follow/follow.go (abridged)
package follow

import (
    "embed"

    "github.com/GoMudEngine/GoMud/internal/events"
    "github.com/GoMudEngine/GoMud/internal/plugins"
    "github.com/GoMudEngine/GoMud/internal/rooms"
    "github.com/GoMudEngine/GoMud/internal/scripting"
    "github.com/GoMudEngine/GoMud/internal/users"
)

//go:embed files/*
var files embed.FS

type followId struct {
    userId        int
    mobInstanceId int
}

type FollowModule struct {
    plug      *plugins.Plugin
    followed  map[followId][]followId // who is followed -> list of followers
    followers map[followId]followId   // follower -> who they follow
}

func init() {
    f := FollowModule{
        plug:      plugins.New("follow", "1.0"),
        followed:  make(map[followId][]followId),
        followers: make(map[followId]followId),
    }

    if err := f.plug.AttachFileSystem(files); err != nil {
        panic(err)
    }

    // Admin page.
    f.plug.Web.AdminPage("Config", "follow-config", "html/admin/follow-config.html",
        true, "Modules", "Follow",
        "Configure follow behavior and allowed follow distances.",
        "Follow system allowing players and NPCs to trail behind another character.", nil)

    // Both a player command and a mob AI command.
    f.plug.AddUserCommand("follow", f.followUserCommand, true, false)
    f.plug.AddMobCommand("follow", f.followMobCommand, true)

    // Expose a function to JavaScript scripts.
    // Callable in scripts as: modules.follow.GetFollowers(actor)
    f.plug.AddScriptingFunction("GetFollowers", f.Scripting_GetFollowers)

    // Multiple event listeners — MobIdle uses events.First to intercept
    // before the built-in mob AI runs.
    events.RegisterListener(events.RoomChange{}, f.roomChangeHandler)
    events.RegisterListener(events.PlayerDespawn{}, f.playerDespawnHandler)
    events.RegisterListener(events.MobDeath{}, f.onMobDeath)
    events.RegisterListener(events.PlayerDeath{}, f.onPlayerDeath)
    events.RegisterListener(events.MobIdle{}, f.idleMobHandler, events.First) // intercept first
    events.RegisterListener(events.PartyUpdated{}, f.onPartyChange)
    events.RegisterListener(events.NewRound{}, f.onNewRound)
}

// Scripting bridge — receives a ScriptActor from JavaScript.
func (f *FollowModule) Scripting_GetFollowers(target scripting.ScriptActor) []*scripting.ScriptActor {
    // ... return follower actors ...
    return nil
}

func (f *FollowModule) followUserCommand(rest string, user *users.UserRecord, room *rooms.Room, flags events.EventFlag) (bool, error) {
    // ... follow logic for players ...
    return true, nil
}

func (f *FollowModule) idleMobHandler(e events.Event) events.ListenerReturn {
    // Return events.Cancel to suppress the built-in mob AI for this mob.
    // Return events.Continue to let the default AI run after this handler.
    return events.Continue
}

Using events.First on MobIdle lets your module run its own AI logic and optionally cancel the built-in mob AI by returning events.Cancel. This is the correct pattern for any module that wants to drive mob behaviour.

Publishing Your Module to the Community Registry

The community module registry lives in a separate GitHub repository: github.com/GoMudEngine/GoMud-Modules. Adding your module there makes it discoverable and installable by anyone using the module manager. The process is a standard GitHub pull request.

Step 1 — Package your module

The registry requires a .tar.gz archive of your module directory and its SHA256 checksum. Use the package subcommand:

terminal
go run . module package <your-module-name>

This creates <your-module-name>.tar.gz in the current directory and prints its SHA256 hash. Keep both — you will need them in the next steps.

Step 2 — Host the archive

Upload the .tar.gz file somewhere publicly accessible over HTTPS. Common options include:

  • A GitHub release on your module's own repository (recommended).
  • Any static file host or object storage bucket that serves raw files.

Copy the direct download URL. It must point to the raw archive file, not a web page.

Step 3 — Fork the GoMud-Modules repository

Go to github.com/GoMudEngine/GoMud-Modules and click Fork to create your own copy of the repository.

Step 4 — Add your entry to module-registry.yaml

In your fork, open module-registry.yaml and append a new entry to the modules list. Every field is required:

module-registry.yaml
modules:
  # ... existing entries ...

  - name: your-module-name
    description: A short, one-line description of what the module does.
    version: "1.0.0"
    author: YourGitHubUsername
    url: https://github.com/you/your-module/releases/download/v1.0.0/your-module-name.tar.gz
    sha256: <paste the SHA256 printed by module package>
FieldDescription
nameLowercase letters, digits, and hyphens only (e.g. my-module). Must match the module's directory name under modules/.
descriptionOne concise sentence shown in module list output.
versionSemantic version string (e.g. "1.0.0"). Increment this on each update.
authorYour name or GitHub username.
urlDirect HTTPS URL to the .tar.gz archive. Must be stable and publicly accessible.
sha256Lowercase hex SHA256 of the archive file. Produced by module package.

Step 5 — Open a pull request

Commit your change to module-registry.yaml in your fork, then open a pull request against the master branch of GoMudEngine/GoMud-Modules. Include a brief description of what your module does and a link to its source repository.

To publish an update to an existing entry, increment version, upload the new archive, update url to point to it, and replace sha256 with the new checksum. Then open another pull request with those changes.

Never change the sha256 of an existing version without also changing version. Users who have already installed the module will see a checksum mismatch error if the archive at a given version URL changes.
