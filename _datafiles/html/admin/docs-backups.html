Backup Guide

Protect your world data with automatic backups to Amazon S3.

How Backups Work

GoMud can automatically create compressed archives of your entire world data directory and upload them to a remote storage provider. This protects you against data loss from hardware failure, accidental deletions, or bad configuration changes.

📦
Compressed Archives
World data is packaged into a single .tar.gz file containing all rooms, mobs, items, users, quests, and plugin data.
🔒
Consistent Snapshots
The server briefly pauses game processing while creating the archive to ensure a consistent point-in-time snapshot.
☁️
Cloud Storage
Completed archives are uploaded to Amazon S3, keeping backups safe even if the server machine is lost.

The backup process follows these steps:

  1. The server acquires the global lock, briefly pausing all game activity.
  2. All files in your world data directory are compressed into a .tar.gz archive in memory.
  3. The server releases the lock and resumes normal operation.
  4. The archive is uploaded to the configured S3 bucket in the background.

The server lock is only held during the compression step. The S3 upload happens after the lock is released, so players experience only a brief pause—typically under a second for most world sizes.

Backup Schedule

Automatic backups can be configured to run on one of three schedules. All scheduled backups run at 3:00 AM server local time to minimize impact on players.

ScheduleConfig ValueWhen It Runs
Nightly nightly Every day at 3:00 AM
Weekly weekly Every Sunday at 3:00 AM
Monthly monthly First day of each month at 3:00 AM
Never never No automatic backups

You can change the schedule at any time through the Config page or the Config Wizard. The server checks the configuration each cycle, so changes take effect without a restart.

Manual Downloads

Admin users can download a backup at any time from the admin home page. The Download World Backup button creates a fresh archive and sends it directly to your browser.

Manual downloads are only available to admin users, not moderators. The server will be briefly unresponsive while the archive is created.

Amazon S3 Setup Guide

Amazon S3 (Simple Storage Service) is a cloud storage service that lets you store files reliably and inexpensively. GoMud uploads backup archives to an S3 "bucket"—a named container for files in your AWS account.

This guide walks you through every step from creating an AWS account to seeing your first successful backup upload. No prior AWS experience is required.

Cost: S3 storage costs roughly $0.023 per GB/month in most regions. A typical GoMud world backup is well under 100 MB, so monthly storage costs are usually pennies. There is also a generous free tier for the first 12 months of a new AWS account.

Step 1 — Create an AWS Account

If you already have an AWS account, skip to Step 2.

  1. 1
    Go to the AWS sign-up page

    Visit aws.amazon.com and click Create an AWS Account.

  2. 2
    Enter your details

    Provide an email address, choose an account name, and set a password. You will also need to provide a payment method (credit/debit card) for verification, though you will not be charged unless you exceed free-tier limits.

  3. 3
    Choose the Free tier support plan

    Select the Basic (Free) support plan. This is sufficient for backup storage.

  4. 4
    Sign in to the AWS Console

    Once your account is ready, sign in at console.aws.amazon.com.

Step 2 — Create an S3 Bucket

A bucket is where your backup files will be stored. Each bucket has a globally unique name.

  1. 1
    Open the S3 console

    In the AWS Console, search for S3 in the top search bar and click on it.

  2. 2
    Click "Create bucket"

    You'll see the bucket creation form.

  3. 3
    Configure the bucket
    • Bucket name: Choose a unique name, e.g. my-gomud-backups
    • AWS Region: Pick a region close to your server for faster uploads. Note this region name (e.g. us-east-1)—you'll need it for the GoMud configuration.
  4. 4
    Keep the default settings

    Leave Block all public access enabled (checked). Your backups should be private. Leave versioning and encryption at their defaults.

  5. 5
    Click "Create bucket"

    Your bucket is now ready.

Optional: You can set up a Lifecycle Rule to automatically delete old backups. In the bucket settings, go to Management → Lifecycle rules and create a rule to expire objects after a set number of days (e.g. 90 days).

Step 3 — Create an IAM User

IAM (Identity and Access Management) lets you create a dedicated user account for GoMud with only the permissions it needs. Never use your root AWS credentials for applications.

  1. 1
    Open the IAM console

    Search for IAM in the AWS Console search bar and click on it.

  2. 2
    Go to Users → Create user

    In the left sidebar, click Users, then click the Create user button.

  3. 3
    Set the user name

    Enter a descriptive name like gomud-backup. Do not check "Provide user access to the AWS Management Console"—this user only needs programmatic access.

  4. 4
    Click Next through permissions

    Skip the permissions step for now—we'll attach a custom policy in the next step. Click Next, then Create user.

Step 4 — Set Permissions

Now we'll give the IAM user permission to upload files to your specific bucket and nothing else. This follows the AWS security best practice of least privilege.

  1. 1
    Click on the user you just created

    From the IAM Users list, click on gomud-backup to open its detail page.

  2. 2
    Go to Permissions → Add permissions → Create inline policy

    Click the Permissions tab, then Add permissions, then choose Create inline policy.

  3. 3
    Switch to the JSON editor

    Click the JSON tab and replace the contents with the policy below. Change my-gomud-backups to your actual bucket name.

IAM Policy JSON
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": "s3:PutObject",
            "Resource": "arn:aws:s3:::my-gomud-backups/*"
        }
    ]
}
  1. 4
    Review and create

    Give the policy a name like GoMudBackupUpload and click Create policy.

This policy only allows s3:PutObject—the user can upload files but cannot read, list, or delete anything. This is the minimum permission GoMud needs for backups.

Step 5 — Get Access Keys

Access keys are how GoMud authenticates with AWS. They consist of two parts: an Access Key ID (starts with AKIA) and a Secret Access Key (a longer random string).

  1. 1
    Open the user's Security credentials tab

    From the IAM user detail page for gomud-backup, click the Security credentials tab.

  2. 2
    Create access key

    Scroll to Access keys and click Create access key. Choose the Application running outside AWS use case, then click Next and Create access key.

  3. 3
    Copy both values

    You will see the Access key ID and Secret access key. Copy both now—the secret key is only shown once. Store them securely.

Keep your Secret Access Key safe. It cannot be retrieved later—if you lose it, you'll need to create a new access key pair. Never share it publicly or commit it to version control.

Step 6 — Configure GoMud

Now enter the S3 details into GoMud. You have two options: the admin panel or environment variables.

Option A: Admin Panel

Go to the Config Wizard and select the Backups section, or edit the values directly on the Config page. Set the following values:

Config KeySet To
Backup.Schedule Your preferred frequency: nightly, weekly, or monthly
Backup.S3.Enabled true
Backup.S3.Bucket Your bucket name, e.g. my-gomud-backups
Backup.S3.Region The AWS region of your bucket, e.g. us-east-1
Backup.S3.Prefix Optional folder prefix, e.g. backups/myworld
Backup.S3.AccessKey The Access Key ID from Step 5
Backup.S3.SecretKey The Secret Access Key from Step 5

Option B: Environment Variables

For production deployments, it is more secure to pass the AWS credentials as environment variables rather than storing them in the config file. Set these before starting GoMud:

Shell
export BACKUP_S3_ACCESS_KEY="AKIAIOSFODNN7EXAMPLE"
export BACKUP_S3_SECRET_KEY="wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"

When environment variables are set, they override the corresponding config file values. The remaining settings (Bucket, Region, Prefix, Schedule, and Enabled) are still configured through the admin panel or config file.

Option C: Config File

You can also set these values directly in config-overrides.yaml:

config-overrides.yaml
Backup:
  Schedule: nightly
  S3:
    Enabled: true
    Bucket: my-gomud-backups
    Region: us-east-1
    Prefix: backups/myworld
    AccessKey: AKIAIOSFODNN7EXAMPLE
    SecretKey: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY

If you store credentials in the config file, make sure the file is not accessible to untrusted users. Environment variables are the recommended approach for production.

Step 7 — Verify It Works

The easiest way to verify your configuration is to trigger a manual backup:

  1. 1
    Download a backup from the admin panel

    Go to the admin home page and click Download World Backup. If the download succeeds, the compression step is working.

  2. 2
    Check the server logs

    After the next scheduled backup runs, look for log entries like:

Expected log output
INFO  Backup  action="scheduled backup starting"
INFO  Backup  action="world archive created"  filename="world-backup-2026-05-30-030000.tar.gz"  size=4521983
INFO  Backup  action="S3 upload complete"  filename="world-backup-2026-05-30-030000.tar.gz"
  1. 3
    Check your S3 bucket

    Open the S3 console in AWS, navigate to your bucket, and confirm the backup file is there.

If you don't want to wait for the next scheduled run, you can temporarily set the schedule to nightly—the scheduler will pick it up within an hour and queue the next backup for 3:00 AM.

Troubleshooting

SymptomLikely CauseSolution
Log shows S3 access key and secret key must be configured Missing or empty credentials Set Backup.S3.AccessKey and Backup.S3.SecretKey via the admin panel or environment variables.
Log shows S3 upload: AccessDenied IAM policy is missing or uses the wrong bucket name Verify the IAM policy grants s3:PutObject on arn:aws:s3:::YOUR-BUCKET/*. Ensure the bucket name matches exactly.
Log shows S3 upload: NoSuchBucket Wrong bucket name or region Double-check Backup.S3.Bucket and Backup.S3.Region. The region must match where the bucket was created.
Log shows S3 upload: InvalidAccessKeyId Access key is incorrect or deactivated Verify the access key ID is correct. Check in IAM that the key is in Active status.
Log shows S3 upload: SignatureDoesNotMatch Secret key is incorrect Regenerate the access key pair in IAM and update the configuration.
No backup log entries at all Schedule is disabled or empty Set Backup.Schedule to nightly, weekly, or monthly. Use never to explicitly disable.
Backup created but not uploaded Backup.S3.Enabled is false Set Backup.S3.Enabled to true.
