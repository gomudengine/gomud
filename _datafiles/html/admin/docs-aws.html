Hosting on Amazon AWS

A complete beginner-friendly guide to launching GoMud on an Amazon EC2 virtual server.

What You Will Build

This guide walks you through renting a virtual Linux server on Amazon Web Services (AWS), configuring it so players can connect, and running GoMud on it. By the end you will have a server that starts automatically when the machine boots and is reachable over the internet.

☁
EC2 Instance
A virtual Linux machine hosted in an AWS data center. You pay only for what you use — there is no physical hardware to buy.
🔒
Security Group
AWS’s built-in firewall. Controls which numbered ports (network “doors”) players can reach and which are blocked.
🔑
SSH Key Pair
A pair of cryptographic files used instead of a password. You keep one half; the server holds the other. Only someone with your half can log in.
🔄
Swap File
A file on disk that acts as overflow memory when RAM (the server’s fast working memory) fills up. Prevents crashes during Go compilation. Set up in Step 10.

No prior AWS experience needed. Every step is explained in plain language. Where a choice exists between options, this guide explains the trade-offs so you can decide what is right for your situation.

Create an AWS Account

If you do not already have an AWS account, go to aws.amazon.com and click Create an AWS Account. You will need:

  • An email address
  • A credit or debit card (AWS will charge a small verification amount, typically refunded)
  • A phone number for identity verification

AWS offers a Free Tier for new accounts. The t2.micro and t3.micro instance types are free for 750 hours per month for the first 12 months. That is enough to run a small MUD server at no cost while you are getting started.

Set up billing alerts immediately after creating your account. Go to Billing → Budgets and create a budget with a $5 or $10 monthly threshold. AWS will email you if you are about to be charged unexpectedly. This is a safety net against accidentally leaving resources running.

Step 1 — Choose a Region

AWS operates data centers in many locations around the world, called regions. When you launch a server, you pick which region it lives in. This affects:

  • Latency: Players connect faster when the server is geographically close to them.
  • Price: Prices vary slightly between regions. us-east-1 (Northern Virginia) is typically the cheapest.
  • Free Tier eligibility: Free Tier applies per region, so it does not matter which one you pick for that purpose.

The region selector is the dropdown in the top-right corner of the AWS console, showing something like N. Virginia or us-east-1. Pick the region closest to where most of your players will be, or just leave it as the default (us-east-1) if you are unsure.

Once you pick a region, stick with it. Resources in one region cannot directly see resources in another region, so keeping everything in the same region avoids confusion.

Step 2 — Open the EC2 Launch Wizard

  1. 1
    Go to the EC2 service

    In the AWS console, type EC2 in the search bar at the top and click EC2 in the results. EC2 stands for “Elastic Compute Cloud”—it is the AWS service for renting virtual machines.

  2. 2
    Click “Launch instance”

    On the EC2 dashboard, click the orange Launch instance button. This opens a multi-step form where you configure your new server.

  3. 3
    Give it a name

    In the Name and tags field, type something descriptive like gomud-server. This is just a label for your own reference in the console.

  4. 4
    Choose an AMI (the operating system image)

    An AMI (Amazon Machine Image) is a pre-built snapshot of an operating system that AWS uses to set up your server. Think of it as the installer disc for your server’s OS.

    Under Application and OS Images, select Ubuntu. Choose the latest Ubuntu Server LTS (Long Term Support) version— at the time of writing this is Ubuntu 24.04 LTS. LTS releases receive security updates for five years, making them the safe choice for a server you intend to leave running without constant maintenance.

    Make sure the architecture is set to 64-bit (x86). This means the server uses the standard Intel/AMD processor instruction set that most software is built for. The alternative (ARM/Graviton) is cheaper but requires a different build of Go and is not covered here.

Step 3 — Choose an Instance Type

The instance type determines how much CPU (processing power) and RAM (Random Access Memory — the fast working memory programs use while running) your server has. AWS charges more for larger instance types. Here are the common choices for GoMud:

TypevCPU (virtual cores)RAMBest ForFree Tier
t2.micro 1 1 GB Testing, very small player counts (<10 simultaneous players) Yes (first 12 months)
t3.micro 2 1 GB Small production server; better CPU burst performance than t2. (Burst means it can briefly use extra CPU power when needed, then throttles back.) Yes (first 12 months)
t3.small 2 2 GB Comfortable for most MUD servers; enough RAM to compile GoMud without needing a swap file No (~$15/month)
t3.medium 2 4 GB Larger worlds, many concurrent players, future growth No (~$30/month)

Recommendation: Start with t3.micro if you want to stay on the Free Tier, or t3.small if you want a more comfortable experience. The 1 GB RAM on micro instances is tight for Go compilation—that is why you will set up a swap file later in this guide. A swap file lets the OS use disk space as overflow memory when RAM fills up.

You can resize your instance type later without losing your data. If you start on t3.micro and find it underpowered, you can stop the instance, change the type, and start it again.

Step 4 — Create an SSH Key Pair

SSH (Secure Shell) is the standard way to log in to a remote Linux server and type commands into it from your own computer. The connection is fully encrypted, so nobody watching the network can see what you type.

Instead of a password, AWS uses a key pair. This is a matched pair of very long numbers generated together: a private key (a file that stays on your computer and must never be shared) and a public key (stored on the server). When you connect, SSH proves you have the private key without ever sending it over the network. Only someone with the private key file can log in, even if they know the server’s IP address.

  1. 1
    Click “Create new key pair”

    In the Key pair (login) section, click Create new key pair.

  2. 2
    Name it and choose the format

    Give it a name like gomud-key.

    • Key pair type: Choose RSA. RSA and ED25519 are both strong encryption algorithms; RSA has slightly wider compatibility with older SSH clients, so it is the safer default choice.
    • Private key file format: Choose .pem if you are on Mac or Linux. .pem is the standard format for the built-in SSH tools on those systems. Choose .ppk only if you are using the PuTTY SSH client on Windows. Most modern Windows users should also choose .pem since Windows 10 and later include a built-in SSH client that works with .pem directly.
  3. 3
    Download and save the key file

    Click Create key pair. Your browser will download a file like gomud-key.pem. Save this file somewhere safe. AWS will never let you download it again. If you lose it, you will be locked out of your server and will need to create a new one.

    A good place to keep it is in a folder like ~/.ssh/ on your computer. The ~ (tilde) symbol is a shorthand for your home directory—on Mac it is /Users/yourname, on Linux it is /home/yourname.

  4. 4
    Restrict the key file permissions (Mac/Linux)

    SSH refuses to use a key file that other users on your computer can read. Open a terminal and run:

Terminal (Mac / Linux)
chmod 400 ~/.ssh/gomud-key.pem

This sets the file to read-only for your user only. On Windows, right-click the file, go to Properties → Security, and ensure only your user account has access.

Step 5 — Configure the Security Group (Firewall)

A security group is AWS’s built-in firewall. A firewall sits in front of your server and decides which incoming network connections to allow and which to silently drop. By default, all inbound traffic to your server is blocked. You must explicitly open each port you want to allow.

A port is a numbered channel on a network connection — like different doors into the same building. Every network service listens on a specific port number. For example, web browsers expect port 80 for regular websites and port 443 for secure ones. MUD clients traditionally connect on port 23 (the Telnet port). You open only the ports your server actually needs.

In the Network settings section, click Edit to expand the rules. You will add the following inbound rules:

TypeProtocolPortSourceWhy
SSH TCP 22 My IP Lets you log in to manage the server. Using My IP restricts this to your current IP address, which is much safer than allowing the whole internet.
Custom TCP TCP 23 Anywhere (0.0.0.0/0) The standard Telnet port. Telnet is the original plain-text network protocol that MUD clients use to connect. 0.0.0.0/0 means “allow connections from any IP address on the internet.”
Custom TCP TCP 80 Anywhere (0.0.0.0/0) HTTP — the standard protocol for web pages. Used for the GoMud web client and admin panel over a plain (unencrypted) connection.
Custom TCP TCP 443 Anywhere (0.0.0.0/0) HTTPS — the encrypted version of HTTP. The “S” stands for Secure. Used for the web client and admin panel when TLS (Transport Layer Security, the technology behind the padlock icon in browsers) is configured.

TCP (Transmission Control Protocol) is the network protocol that all of these services use. It is the reliable, connection-oriented way computers send data to each other over the internet. You will always choose TCP here; the alternative (UDP) is for streaming and games that can tolerate dropped packets, which does not apply to a MUD.

Do not set SSH (port 22) to “Anywhere”. Automated bots on the internet constantly scan every IP address for open SSH ports and attempt brute-force logins — meaning they try thousands of username and password combinations per second. Restricting SSH to My IP means only your computer’s current IP address can even attempt to connect. If your home IP address changes (which can happen when your router restarts), you can update this rule in the AWS console under Security Groups.

How to add a rule

  1. 1
    The SSH rule is usually pre-filled

    AWS typically adds an SSH rule automatically. Change its source from Anywhere to My IP by selecting it from the dropdown.

  2. 2
    Add port 23 (Telnet)

    Click Add security group rule. Set Type to Custom TCP, Port range to 23, and Source to Anywhere (0.0.0.0/0). Add a description like GoMud Telnet.

  3. 3
    Add port 80 (HTTP)

    Click Add security group rule again. Set Type to HTTP (AWS will fill in port 80 automatically). Source: Anywhere.

  4. 4
    Add port 443 (HTTPS)

    Click Add security group rule again. Set Type to HTTPS (AWS will fill in port 443 automatically). Source: Anywhere.

GoMud’s default configuration uses port 23 for Telnet and port 80 for the web interface. You can change these in the GoMud config file later. If you use non-standard ports (e.g., 4000 for Telnet), add those ports to the security group instead of or in addition to port 23.

Step 6 — Configure Storage

Under Configure storage, you set the size of the server’s virtual hard drive (called an EBS volume in AWS terminology — Elastic Block Store, which is just AWS’s name for a network-attached disk). The default is usually 8 GB. For GoMud, 20 GB is a comfortable amount that gives you room for the operating system, Go toolchain, GoMud itself, world data, logs, and backups.

The volume type should be gp3 (General Purpose SSD, version 3). An SSD (Solid State Drive) has no moving parts and is much faster than a traditional spinning hard disk, which matters for server responsiveness. gp3 is faster and cheaper than the older gp2 type. AWS may default to gp3 already; if it shows gp2, change it.

The Free Tier includes 30 GB of gp2 storage. If you are staying on the Free Tier, keep the volume at 8 GB or 30 GB maximum to avoid charges. The 8 GB default is workable for GoMud.

Step 7 — Review and Launch

The right-hand Summary panel shows your choices. Review them and click Launch instance. AWS will take about 30–60 seconds to start your server.

Once launched, click View all instances to go to the EC2 instances list. Wait until the Instance state column shows Running and the Status check column shows 2/2 checks passed. This means the server has booted successfully.

In the instance details at the bottom of the page, find the Public IPv4 address. An IP address is the unique numerical label that identifies your server on the internet — similar to a postal address but for network traffic. IPv4 addresses look like four numbers separated by dots, for example 54.123.45.67. Write this down — you will need it to connect via SSH and to tell players where to connect.

The Public IPv4 address changes every time you stop and restart the instance. AWS recycles IP addresses from a shared pool, so each start gets a different one. If you want a permanent, fixed address, you can allocate an Elastic IP from the EC2 console (Network & Security → Elastic IPs) and associate it with your instance. “Elastic” just means it stays yours until you explicitly release it. Elastic IPs are free as long as they are attached to a running instance, but cost a small hourly fee if the instance is stopped (to discourage hoarding unused addresses).

Step 8 — Connect to Your Server via SSH

A terminal (also called a command line or shell) is a text-based window where you type commands directly. On Mac, open Terminal from Applications → Utilities. On Linux, open your system’s terminal application. On Windows 10 or later, open PowerShell from the Start menu (search for “PowerShell”).

Replace YOUR_IP with the Public IPv4 address from the previous step and ~/.ssh/gomud-key.pem with the path to your key file.

Terminal (Mac / Linux / Windows PowerShell)
ssh -i ~/.ssh/gomud-key.pem ubuntu@YOUR_IP

The username is ubuntu because you launched an Ubuntu image. AWS creates this user automatically. You do not use a password; the key file authenticates you.

The first time you connect, SSH will ask:

Expected prompt
The authenticity of host '1.2.3.4 (1.2.3.4)' can't be established.
ED25519 key fingerprint is SHA256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx.
Are you sure you want to continue connecting (yes/no/[fingerprint])?

Type yes and press Enter. SSH is showing you a fingerprint — a short unique identifier derived from the server’s own cryptographic key — and asking whether you trust it. On a server you just created yourself it is safe to accept. SSH saves this fingerprint locally so it can warn you if it ever changes unexpectedly (which could indicate someone is intercepting your connection).

You should now see a prompt like ubuntu@ip-172-31-xx-xx:~$. You are logged in to your server.

Windows users: If you are on Windows 10 or later, PowerShell includes a built-in SSH client. The command above works in PowerShell. If you prefer a graphical SSH client, PuTTY is a popular free option (use the .ppk key format with PuTTY).

Step 9 — Update the System

Before installing anything, update the list of available packages and apply any security patches. This is good practice any time you set up a new Linux server.

Server terminal
sudo apt update && sudo apt upgrade -y

apt is Ubuntu’s package manager — think of it as an app store for command-line software. It downloads and installs programs from Ubuntu’s official repositories (trusted software libraries maintained by the Ubuntu project). sudo stands for “superuser do” and runs the command as root (the administrator account that has permission to change system files). Without sudo, the command would be blocked because regular users cannot modify system software. The -y flag automatically answers “yes” to any confirmation prompts so the upgrade runs without you needing to sit and watch it.

This may take a minute or two. You will see a lot of output scrolling past; that is normal. When it finishes, the prompt returns.

Step 10 — Set Up a Swap File

Your server has a fixed amount of RAM (Random Access Memory) — the fast working memory that programs use while they are running. When a program needs more RAM than is available, the operating system normally terminates it immediately.

Swap space gives the OS a safety valve: it reserves a portion of the (slower) disk as overflow memory. When RAM fills up, the OS moves the least-recently-used data out of RAM and onto the swap area to free up space. This is slower than real RAM, but it prevents the program from being killed outright.

When you compile GoMud (translate its source code into an executable program the computer can run), the Go compiler briefly needs significantly more memory than a 1 GB instance provides. Without swap, the OS kills the compiler mid-way through with a cryptic signal: killed message and the build fails.

A 2 GB swap file is sufficient. Run the following commands one at a time:

Server terminal — create and enable the swap file
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

What each command does:

  • fallocate -l 2G /swapfile — allocates a 2 GB file at /swapfile. This is fast because it reserves the space without writing data.
  • chmod 600 /swapfile — restricts the file so only root can read or write it. Linux requires this before it will use a file as swap.
  • mkswap /swapfile — formats the file with the swap filesystem structure.
  • swapon /swapfile — activates the swap file so the OS starts using it immediately.

Verify it is active:

Server terminal
free -h

You should see a Swap: row with 2.0Gi (or similar) in the total column. The used column will be near zero at this point; swap only gets used when RAM fills up.

Make swap permanent across reboots

The swap file is active now, but it will not survive a reboot unless you register it permanently. Linux reads a file called /etc/fstab (filesystem table) during boot to know which disks and files to mount. Adding the swap file there tells Linux to activate it automatically every time the server starts.

Server terminal
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

This appends one line to /etc/fstab. The fields mean: use /swapfile as swap space, with no mount point (none), of type swap, with standard options (sw), and do not dump or check it on boot (0 0).

The tee -a pattern is used instead of a plain shell redirect (>>) because a redirect runs before sudo takes effect, so it would not have permission to write to a system file. Piping into sudo tee -a routes the output through a command that does run as root.

If you chose a t3.small or larger instance with 2 GB+ RAM, you can skip the swap file. It does not hurt to have it anyway as a safety net, but it is not strictly required for compilation on larger instances.

Step 11 — Install Go

GoMud is written in the Go programming language and must be compiled from source. Compiling means taking the human-readable source code files and translating them into a single executable binary file that the computer can run directly. You only need to do this once (and again after any updates).

Ubuntu’s built-in Go package is often several versions behind the latest release, which can cause build errors. It is safer to install Go directly from the official Go website to get the current stable version.

First, find the latest Go version at go.dev/dl. At the time of writing, the latest stable release is go1.23. Substitute the correct version number in the commands below.

Server terminal — download and install Go
# Download the Go archive (replace 1.23.0 with the current version)
wget https://go.dev/dl/go1.23.0.linux-amd64.tar.gz

# Remove any previous Go installation and extract the new one
sudo rm -rf /usr/local/go
sudo tar -C /usr/local -xzf go1.23.0.linux-amd64.tar.gz

# Clean up the downloaded archive
rm go1.23.0.linux-amd64.tar.gz

Next, add Go to your shell’s PATH. The PATH is a list of directories the shell searches when you type a command. Without adding Go’s directory to PATH, you would have to type the full path (/usr/local/go/bin/go) every time instead of just go:

Server terminal — add Go to PATH
echo 'export PATH=$PATH:/usr/local/go/bin' >> ~/.bashrc
source ~/.bashrc

~/.bashrc is a configuration script that Bash (the default shell on Ubuntu) runs automatically every time you open a new terminal session. Adding the export PATH=... line there means Go will always be on your PATH in every future session. The source command reads and executes a file in the current shell session, so source ~/.bashrc applies the change immediately without requiring you to log out and back in.

Verify Go is installed:

Server terminal
go version

You should see output like go version go1.23.0 linux/amd64.

Step 12 — Download and Build GoMud

You will need git to download the GoMud source code. Git is a version control tool; here you are using it simply to download (or “clone”) a copy of the GoMud code from GitHub onto your server.

Server terminal
sudo apt install -y git make

Now clone the GoMud repository (the hosted collection of source code files) and build it:

Server terminal
# Clone the repository into a folder called gomud
git clone https://github.com/GoMudEngine/GoMud.git gomud

# Enter the directory
cd gomud

# Build the server binary
make build

The make build command compiles GoMud. This will take a few minutes on a small instance, especially the first time. You will see output from the Go compiler scrolling past. When it finishes successfully, the prompt returns without an error.

If the build is killed mid-way with a message like signal: killed, it ran out of memory. Make sure you completed the swap file step above, then try again. The swap file gives the compiler the extra memory it needs.

Step 13 — Run GoMud

Start the server with:

Server terminal (from inside the gomud directory)
make run

You should see log output as GoMud loads its data files and starts listening for connections. Look for a line indicating the server is ready, such as:

Expected log output
INFO  Server listening  telnet=":23"  web=":80"

From your own computer, open a MUD client (or telnet) and connect to YOUR_IP on port 23. You should see the GoMud welcome screen. You can also open a browser and navigate to http://YOUR_IP to reach the web client.

To stop the server, press Ctrl+C in the terminal.

Running with make run directly in your SSH session means the server will stop when you disconnect. The next section shows how to set it up as a background service that keeps running even after you log out.

Step 14 — Auto-Start GoMud on Boot

To keep GoMud running after you disconnect from SSH, and to have it start automatically when the server reboots, you will create a systemd service. Systemd is the init system and process manager built into Ubuntu. It starts system services in the right order when the machine boots, monitors them while they run, and restarts them if they crash. A “service” in systemd terms is just a program that systemd manages, defined by a small configuration file.

  1. 1
    Find the full path to your gomud directory

    Run pwd from inside the gomud directory to see the full path. It will be something like /home/ubuntu/gomud.

  2. 2
    Create the service file

    Run the following command, substituting the correct path if yours differs:

Server terminal — create the systemd service
sudo tee /etc/systemd/system/gomud.service > /dev/null << 'EOF'
[Unit]
Description=GoMud Server
After=network.target

[Service]
Type=simple
User=ubuntu
WorkingDirectory=/home/ubuntu/gomud
ExecStart=/home/ubuntu/gomud/bin/gomud
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF

What the key fields mean:

  • After=network.target — wait until the network is fully up before starting GoMud. Without this, GoMud might start before the network is ready and fail to open its ports (the numbered network channels it listens on for incoming connections).
  • User=ubuntu — run the process as the ubuntu user, not as root. This limits the damage if the process is ever compromised.
  • WorkingDirectory — the directory GoMud expects to run from. It looks for its _datafiles folder relative to this path.
  • ExecStart — the full path to the compiled GoMud binary. The binary is placed in the bin/ subdirectory by make build.
  • Restart=on-failure — if GoMud crashes, systemd will restart it automatically after 5 seconds.
  1. 3
    Enable and start the service
Server terminal
# Reload systemd so it picks up the new service file
sudo systemctl daemon-reload

# Enable the service so it starts automatically on boot
sudo systemctl enable gomud

# Start it now
sudo systemctl start gomud
  1. 4
    Check that it is running
Server terminal
sudo systemctl status gomud

You should see Active: active (running) in green. If it shows failed instead, run sudo journalctl -u gomud -n 50 to see the last 50 lines of output and look for an error message.

To follow the live server logs as they are written, use:

Server terminal — follow live logs
sudo journalctl -u gomud -f

journalctl is the tool for reading systemd’s log journal — the central log store where all service output is collected. The -u gomud flag filters to only GoMud’s output, and -f means “follow” (keep streaming new lines as they appear, like tail -f). Press Ctrl+C to stop following. The server keeps running.

Managing the service

TaskCommand

Troubleshooting

SymptomLikely CauseSolution
SSH connection refused or times out Security group does not allow port 22 from your IP, or the instance is not running Check the security group inbound rules in the AWS console. Make sure port 22 is open for your IP. Verify the instance state is Running.
Permission denied (publickey) when connecting via SSH Wrong key file, wrong username, or key file permissions are too open Make sure you are using the correct .pem file and the username ubuntu. Run chmod 400 ~/.ssh/gomud-key.pem to fix permissions.
Build is killed with signal: killed Out of memory during compilation Set up the swap file as described in Step 10, then run make build again.
Players cannot connect on port 23 Port 23 is not open in the security group, or GoMud is not running Check the security group has a rule for port 23 with source 0.0.0.0/0. Run sudo systemctl status gomud to verify the service is running.
Admin panel at http://YOUR_IP is not reachable Port 80 is not open in the security group Add an HTTP (port 80) inbound rule in the security group.
IP address changed after restarting the instance AWS assigns a new public IP each time a non-Elastic-IP instance starts Allocate an Elastic IP from the EC2 console and associate it with your instance for a permanent address.
GoMud service does not start after reboot Service not enabled, or binary path in the service file is wrong Run sudo systemctl enable gomud to enable auto-start. Verify the ExecStart path in /etc/systemd/system/gomud.service matches where make build placed the binary.
