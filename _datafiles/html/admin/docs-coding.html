Coding Reference

Engine extension points and patterns for Go module authors.

Hooks

A hook is a named extension point in the engine where modules can inject behavior without modifying core code. Hooks are typed, synchronous, and safe for concurrent registration from multiple modules.

Hooks use a value-threading model: each registered handler receives the current value, may modify any field, and returns the (possibly modified) value. Handlers are called in registration order. The final value after all handlers have run is returned to the engine.

If no handlers are registered, the hook is a no-op: the original value passes through unchanged. You never need to guard against a hook being unused.

The generic type is declared in internal/util/hook.go:

internal/util/hook.go
type Hook[T any] struct { ... }

// Register adds a handler. Handlers are called in registration order.
func (h *Hook[T]) Register(fn func(T) T)

// Fire calls every registered handler in order and returns the final value.
func (h *Hook[T]) Fire(data T) T

Usage Pattern

All hooks follow the same three-step pattern: the engine defines the hook and its data type, the engine fires the hook at the right moment, and modules register handlers during their init() or setup phase.

Defining a hook (engine side)

internal/rooms/roomdetails.go
// Declare the hook as a package-level variable.
var OnRoomLook util.Hook[RoomTemplateDetails]

Firing the hook (engine side)

internal/rooms/roomdetails.go
// At the end of GetDetails, pass the fully-populated struct through the hook.
return OnRoomLook.Fire(details)

Registering a handler (module side)

modules/mymodule/mymodule.go
func (m *MyModule) setup() {
    rooms.OnRoomLook.Register(m.onRoomLook)
}

func (m *MyModule) onRoomLook(d rooms.RoomTemplateDetails) rooms.RoomTemplateDetails {
    // Only act on rooms that have the "fishing" tag.
    for _, tag := range d.Tags {
        if tag == "fishing" {
            d.Alert(`You can fish here! Type fish to cast a line.`)
            break
        }
    }
    return d
}

Always return the value — even when your handler makes no changes. Failing to return causes the value to be zeroed out for every subsequent handler in the chain.

Hooks fire on the main server goroutine under the MUD lock. Keep handlers fast. Never block, sleep, or perform I/O that could stall the round loop.

rooms.OnRoomLook

OnRoomLook internal/rooms

Fired at the very end of rooms.GetDetails, after the engine has fully populated the room view for a specific player. Handlers can read or modify any field before the details are handed off to the template renderer. This is the primary hook for adding room-level UI — custom alerts, contextual descriptions, or additional visible entities.

Fires once per look command (or any system that calls rooms.GetDetails), once per viewing player.

What you can do

  • Append alerts below the room description using d.Alert(...)
  • Prepend, append, or replace d.Title or d.Description
  • Add entries to d.VisibleMobs or d.VisiblePlayers for custom entities
  • Inspect d.Tags to conditionally activate behavior based on room tags
  • Read d.Character or d.UserId to personalize the view per player

Data type: rooms.RoomTemplateDetails

FieldTypeAccessDescription
UserIdintreadID of the player viewing the room.
Character*characters.CharacterreadThe character record of the viewing player. Useful for personalizing output.
ZonestringreadZone name the room belongs to.
Titlestringread/writeRoom title shown at the top of the look output. May contain ANSI tags.
Descriptionstringread/writeRoom description body. May contain ANSI tags and newlines.
Tags[]stringreadRoom tags. Use these to opt in to module behavior (e.g. "fishing", "storage").
RoomAlerts[][]stringread/writeGrouped alert lines rendered below the description. Use d.Alert(...) to append.
VisiblePlayers[]stringread/writeFormatted player name strings visible in the room.
VisibleMobs[]stringread/writeFormatted mob name strings visible in the room.
VisibleCorpses[]stringread/writeFormatted corpse strings visible in the room.
VisibleExitsmap[string]exit.RoomExitread/writeExits shown to the player. Keyed by direction/name.
TemporaryExitsmap[string]exit.TemporaryRoomExitread/writeTemporary exits (portals, etc.) visible to the player.
IsDarkboolreadWhether the room's biome is flagged as dark.
IsNightboolreadWhether it is currently night in the game world.
ShowPvpboolread/writeWhether a PvP indicator is shown for this room.
TrackingStringstringread/writeTracking status line shown when the player is following a target.
RoomSymbolstringread/writeSingle-character map symbol for this room.
RoomLegendstringread/writeMap legend label for this room.
Nouns[]stringreadInteractable nouns defined on the room.

Helper method: Alert

Rather than appending to RoomAlerts directly, use the Alert helper. Each call creates one alert group (a bordered block). Multiple string arguments or a string containing \n become separate lines within the same group.

example
// Single-line alert
d.Alert(`A chest is here! Type open chest to loot it.`)

// Multi-line alert (rendered as one bordered block)
d.Alert(
    `Auction House`,
    `Current lot: Sword of Dawn`,
)

Example registration

modules/mymodule/mymodule.go
rooms.OnRoomLook.Register(func(d rooms.RoomTemplateDetails) rooms.RoomTemplateDetails {
    for _, tag := range d.Tags {
        if tag == "fishing" {
            d.Alert(`Fishing spot. Type fish to cast a line.`)
            break
        }
    }
    return d
})

characters.OnGetFormattedName

OnGetFormattedName internal/characters

Fired at the end of the internal getFormattedName function, after the engine has assembled the full display name for a character (player or mob). Handlers can modify any part of the name before it is used in room descriptions, combat messages, or any other context where character names are rendered.

Fires every time a character name is rendered — which can be very frequent. Keep handlers extremely lightweight.

What you can do

  • Set f.Title to append a title after the name (e.g. "Mayor of Frostfang")
  • Append to f.Adjectives to add a visible status indicator
  • Set f.QuestAlert to show the quest star prefix
  • Inspect f.Type to distinguish mob names ("mob") from player names ("user")

Data type: characters.FormattedName

FieldTypeAccessDescription
NamestringreadThe raw character name.
Typestringread"mob" or "user". Determines the ANSI color alias used when rendering.
SuffixstringreadOptional ANSI alias suffix (e.g. "dead"). Combined with Type to form the color alias.
Titlestringread/writeOptional title appended after the name, separated by a comma. Set this to add a role or rank.
Adjectives[]stringread/writeStatus indicators shown in parentheses after the name (e.g. "sleeping", "poisoned"). Append to add custom states.
UseShortAdjectivesboolreadWhether to use short symbol forms for adjectives (player preference).
QuestAlertboolread/writeWhen true, a quest star ★ is prepended to the name.
PetNamestringread/writeIf set, rendered as "[Name] and [PetName]".

Example registration

modules/elections/elections.go
characters.OnGetFormattedName.Register(func(f characters.FormattedName) characters.FormattedName {
    if f.Type != "user" {
        return f
    }
    if title := m.getTitleForUser(f.Name); title != "" {
        f.Title = title // e.g. "Mayor of Frostfang"
    }
    return f
})

usercommands.OnShopList

OnShopList internal/usercommands

Fired once per seller, before the shop inventory is rendered (for the list command) or matched against a purchase (for the buy command). Handlers receive the effective stock for that seller and can add, remove, or reprice items before the player sees or acts on them.

Fires on both list and buy. Use r.IsBuy to distinguish the two if needed.

What you can do

  • Append to r.Stock to inject additional items into any shop
  • Filter r.Stock to remove items from specific sellers
  • Modify Price on individual ShopItem entries to apply discounts or surcharges
  • Use r.SellerMob or r.SellerUser to target a specific NPC or player shop

Data type: usercommands.ShopListRequest

FieldTypeAccessDescription
Stockcharacters.Shopread/writeThe effective inventory for this seller. Add, remove, or modify entries here.
Buyer*users.UserRecordreadThe player browsing or buying. Always non-nil.
SellerMob*mobs.MobreadThe NPC seller, or nil if the seller is a player.
SellerUser*users.UserRecordreadThe player seller, or nil if the seller is an NPC.
Room*rooms.RoomreadThe room where the transaction is taking place. Always non-nil.
IsBuyboolreadtrue when called from the buy command, false from list.

Example registration

modules/mymodule/mymodule.go
usercommands.OnShopList.Register(func(r usercommands.ShopListRequest) usercommands.ShopListRequest {
    if r.SellerMob == nil || r.SellerMob.MobId != mySpecialVendorId {
        return r
    }
    // Inject a limited-time item into this vendor's stock.
    r.Stock = append(r.Stock, characters.ShopItem{ItemId: 42, Price: 500})
    return r
})

usercommands.OnShopListRendered

OnShopListRendered internal/usercommands

Fired once per seller after all shop tables have been sent to the buyer, but only during the list command — it is not fired during buy. Use this hook to append supplementary text after the listing, such as zone tax notices, loyalty point balances, or promotional messages.

Only fires on list, never on buy. Use OnShopList if you also need to intercept purchases.

What you can do

  • Call r.Buyer.SendText(...) to append custom text after the shop table
  • Display tax rates, membership discounts, or special offers
  • Show the buyer their current balance in a module-specific currency

Data type: usercommands.ShopListRequest

Same structure as OnShopList. Stock reflects the final inventory that was rendered. IsBuy is always false here.

Example registration

modules/elections/elections.go
usercommands.OnShopListRendered.Register(func(r usercommands.ShopListRequest) usercommands.ShopListRequest {
    if !m.roomIsInElectionZone(r.Room) {
        return r
    }
    taxRate := m.getTaxRate()
    r.Buyer.SendText(fmt.Sprintf(
        `Zone tax: %d%% — applied at checkout.`, taxRate,
    ))
    return r
})

usercommands.OnInsufficientFunds

OnInsufficientFunds internal/usercommands

Fired during the buy command when the buyer's gold is less than the purchase price. Handlers can send a custom message to the buyer and set Handled to true to suppress the default "You don't have enough gold" message. This is useful for modules that provide alternative currencies or financing mechanics.

Only fires when the buyer cannot afford the item. Setting Handled = true suppresses the default error message but does not complete the purchase.

What you can do

  • Send a custom, context-aware "insufficient funds" message
  • Set Handled = true to replace the default engine message with your own
  • Suggest alternatives (e.g. "You need 50 more gold — try the bank on the west side")

Data type: usercommands.InsufficientFundsRequest

FieldTypeAccessDescription
Buyer*users.UserRecordreadThe player attempting the purchase.
SellerMob*mobs.MobreadThe NPC seller, or nil if the seller is a player.
SellerUser*users.UserRecordreadThe player seller, or nil if the seller is an NPC.
Room*rooms.RoomreadThe room where the failed purchase occurred.
GoldintreadThe buyer's current gold at the time of the attempt.
PriceintreadThe full price required (may be tax-inclusive).
Handledboolread/writeSet to true to suppress the default "not enough gold" message from the engine.

Example registration

modules/mymodule/mymodule.go
usercommands.OnInsufficientFunds.Register(func(r usercommands.InsufficientFundsRequest) usercommands.InsufficientFundsRequest {
    shortfall := r.Price - r.Gold
    r.Buyer.SendText(fmt.Sprintf(
        `You are %d gold short. The nearest bank is to the east.`,
        shortfall,
    ))
    r.Handled = true
    return r
})

suggestions.OnAutoComplete

OnAutoComplete internal/suggestions

Fired after the engine's built-in autocomplete logic has run. Modules register handlers here to contribute tab-completion suggestions for their own commands. Each handler receives the accumulated results from all previous handlers and should append to — not replace — r.Results.

Fires on every tab-press when the player has typed at least one argument. Results are sorted shortest-first before being sent to the client.

What you can do

  • Provide completions for module-added commands
  • Return early (without modifying r.Results) when r.Cmd is not your command
  • Use r.Parts for multi-word completions (e.g. sub-commands or named arguments)
  • Use r.UserId to personalize completions based on the player's state

Important: result format

Each entry in Results is the suffix to append to the current input, not a full replacement string. If the player has typed fish c and you want to suggest cast, append "ast" — not "cast".

Data type: suggestions.AutoCompleteRequest

FieldTypeAccessDescription
UserIdintreadID of the player requesting completions.
CmdstringreadThe resolved command name after alias expansion (e.g. "fish"). Always check this first and return early if it is not your command.
Parts[]stringreadThe original input split on whitespace. Parts[0] is the raw command token (before alias expansion). Parts[1:] are the arguments typed so far.
Results[]stringread/writeAccumulated completion suffixes. Append your suggestions here; do not replace the slice.

Example registration

modules/fishing/fishing.go
suggestions.OnAutoComplete.Register(func(r suggestions.AutoCompleteRequest) suggestions.AutoCompleteRequest {
    if r.Cmd != "fish" {
        return r // not our command — leave results untouched
    }

    subcommands := []string{"cast", "release", "inventory"}
    typed := ""
    if len(r.Parts) > 1 {
        typed = strings.ToLower(r.Parts[1])
    }

    for _, sub := range subcommands {
        if strings.HasPrefix(sub, typed) {
            r.Results = append(r.Results, sub[len(typed):])
        }
    }
    return r
})
